Announcing "The Invisible Users"

A Guide to Building Websites That Work for AI Agents and Everyone Else

By Tom Cranstoun • 15 minute read

The Problem We're Solving

AI agents are visiting your website right now. They're researching products on behalf of users, comparing pricing across competitors, attempting to complete purchases, and filling out contact forms. And a significant percentage of them are failing silently.

These failures don't appear in your analytics. You never see the bounce. The conversion simply never happens.

The patterns that break AI agents also break human users. Screen reader users have been struggling with these same issues for years. People with cognitive disabilities, anyone processing pages sequentially, users on slow connections, anyone who can't perceive visual state changes - they've all encountered these barriers.

Now there's commercial pressure to fix them.

What Is "The Invisible Users"?

"The Invisible Users: Designing the Web for AI Agents and Everyone Else" is a practical guide to building websites that work for both human users and AI agents. It's not speculation about a distant future - this is happening right now, affecting real businesses and real users.

The book covers:

  • Technical patterns that fail - Specific examples of what breaks and why
  • Business implications - Economics, competitive dynamics, and strategic decisions
  • Solution patterns - Concrete implementations that work for everyone
  • Production-ready code - HTML, JavaScript, React, WordPress, Next.js examples
  • Testing strategies - Playwright test suite for validating agent compatibility

What Makes This Book Different?

Real Examples, Not Speculation

The tour operator story in Chapter 1 is real. I asked an AI assistant to compare tour operators. It recommended Company A over Company B because Company A's website used pagination - the agent saw only Day 1 of a multi-day tour and concluded that was the entire offering.

I caught the error. How many people wouldn't?

Technical Accuracy with Practical Implications

This isn't theoretical. Every code example is production-ready. Every pattern has been tested. Every recommendation considers real-world constraints.

Sequential Reading with Clear Dependencies

You don't need to read the entire book. Each chapter builds on previous concepts, but you can jump to the sections relevant to your role. The book acknowledges what you need to know before diving into specific topics.

Focus on Universal Patterns

The patterns work for the full spectrum of AI agents:

  • CLI agents running locally (Claude Code, Cline)
  • Lightweight agents on user devices
  • Server-based agents accessing remotely (ChatGPT, Claude API)
  • Browser automation agents (Playwright, Selenium)
  • Browser extension assistants with session inheritance

You can't design for one agent type without excluding others. The guidance focuses on universal compatibility.

Industry projections suggest 5-15% of web traffic will be AI agents by 2026. For e-commerce sites, the percentage may be higher as agents become the primary interface for product research and price comparison.

First-Mover Advantage Window

The competitive advantage window is estimated at 12-18 months. Early adopters capture agent traffic that competitors lose. Once best practices become standard, the advantage disappears.

We've known about these problems for years. Accessibility advocates have been explaining them. But there wasn't enough commercial pressure to fix them. Now there is.

When agents make purchasing decisions worth millions, businesses pay attention. When conversion funnels have invisible drop-offs, there's budget to fix them. When competitors' sites work for agents and yours doesn't, the commercial incentive becomes obvious.

The Principles

The book advocates for five core principles that benefit all users:

  1. Clear State - Use HTML attributes, not just CSS classes. State should be explicit in the DOM, readable by both agents and assistive technologies.
  2. Persistent Errors - No toast notifications that vanish. Errors stay visible in the page content where they can be read and acted upon.
  3. Complete Information - Show full pricing. Avoid forced pagination. Make content accessible without requiring multiple interactions.
  4. Semantic Structure - Use proper HTML elements. Implement Schema.org structured data. Follow established standards.
  5. Explicit Feedback - Confirm actions clearly in page content. Don't rely on visual cues alone. Make success and failure states unambiguous.
